# ARSW-Lab04

# Summary

This lab builds a REST API for blueprint management using Java 21 and Spring Boot 3.3.x. Best architectural practices are applied progressively: layer separation, Port/Adapter and Strategy patterns, interchangeable persistence (in-memory / PostgreSQL), uniform responses with `ApiResponse<T>`, OpenAPI documentation, and processing filters activatable via Spring profiles.

---

### 1. Codebase Familiarization
- Review the `model` package with the `Blueprint` and `Point` classes.
- Understand the `persistence` layer with `InMemoryBlueprintPersistence`.
- Analyze the `services` layer (`BlueprintsServices`) and the `BlueprintsAPIController` controller.

**`model` layer**

- `Point` is a Java *record* with two fields: `x` and `y`, representing a 2D plane coordinate.
- `Blueprint` has three attributes: `author` (String), `name` (String), and an internal list of `Point`. Its identity (equals/hashCode) is defined by the `author + name` combination, which allows it to be used as a unique key. The point list is exposed immutably (`getPoints()` returns `unmodifiableList`); points are added via `addPoint(Point p)`.

**`persistence` layer**

- `BlueprintPersistence` is the **interface (port)** that defines the data access contract: `saveBlueprint`, `getBlueprint`, `getBlueprintsByAuthor`, `getAllBlueprints`, and `addPoint`.
- `InMemoryBlueprintPersistence` is the **adapter** that implements the port using a thread-safe `ConcurrentHashMap<String, Blueprint>`. The map key is the string `author:name`. On startup, it loads three sample blueprints (john/house, john/garage, jane/garden).
- `BlueprintNotFoundException` and `BlueprintPersistenceException` are checked exceptions that signal, respectively, that a blueprint does not exist or that a persistence error occurred (e.g., trying to save a duplicate blueprint).

**`services` layer**

- `BlueprintsServices` injects `BlueprintPersistence` and `BlueprintsFilter` as **interfaces**, not as concrete implementations. This applies the **Strategy pattern** combined with the **Port/Adapter pattern**: the persistence implementation (from in-memory to PostgreSQL) or the active filter can be changed without modifying the service.
- The `getBlueprint()` method is the only one that applies the filter before returning the result; the other methods delegate directly to the persistence layer.

**`filters` layer**

- `BlueprintsFilter` is an interface with a single method `apply(Blueprint) → Blueprint`, making it compatible with lambda expressions.
- `IdentityFilter` is the default implementation: returns the blueprint without any modification.

**`controllers` layer**

- `BlueprintsAPIController` is a `@RestController` mapped to `/api/v1/blueprints` that exposes five endpoints:

| Method | Path | Success Code | Description |
|--------|------|-------------|-------------|
| GET | `/api/v1/blueprints` | 200 | Returns all blueprints |
| GET | `/api/v1/blueprints/{author}` | 200 | Returns all blueprints by an author |
| GET | `/api/v1/blueprints/{author}/{bpname}` | 200 | Returns a specific blueprint (with filter applied) |
| POST | `/api/v1/blueprints` | 201 | Creates a new blueprint |
| PUT | `/api/v1/blueprints/{author}/{bpname}/points` | 202 | Adds a point to an existing blueprint |

The controller uses `BlueprintsServices` as its only dependency, respecting layer separation.

### 2. Migration to PostgreSQL Persistence
- Configure a PostgreSQL database (Docker can be used).
- Implement a new `PostgresBlueprintPersistence` repository to replace the in-memory version.
- Maintain the `BlueprintPersistence` interface contract.

**Design decisions**

The **Port/Adapter pattern** is maintained: the `BlueprintPersistence` interface did not change; only a new implementation was added. The Spring profile (`@Profile("postgres")` / `@Profile("memory")`) decides at startup which adapter to inject, without touching anything else in the application.

**Added structure**

```
persistence/
  exception/
    BlueprintNotFoundException.java      ← exceptions moved to subpackage
    BlueprintPersistenceException.java
  memory/
    InMemoryBlueprintPersistence.java    ← @Profile("memory")
  postgresql/
    entity/
      BlueprintEntity.java               ← JPA entity with UUID as PK
      PointEmbeddable.java               ← @Embeddable, points in "points" table
    mapper/
      BlueprintMapper.java               ← MapStruct: Blueprint ↔ BlueprintEntity
    repository/
      BlueprintJpaRepository.java        ← Spring Data JPA
    PostgresBlueprintPersistence.java    ← @Profile("postgres"), implements the port
```

**Relational model generated by Hibernate**

| Table | Key columns |
|-------|------------|
| `blueprints` | `id` (UUID PK), `author`, `name` (UNIQUE together) |
| `points` | `blueprint_id` (FK → blueprints), `x`, `y` |

Points are stored as `@ElementCollection` in the `points` table, linked by `blueprint_id`.

**Key technical decisions**

- **MapStruct** converts between the domain model (`Blueprint`/`Point`) and JPA entities (`BlueprintEntity`/`PointEmbeddable`), avoiding coupling the domain to JPA.
- The `maven-compiler-plugin` configures `annotationProcessorPaths` with Lombok first and MapStruct second, ensuring Lombok generates getters/setters before MapStruct uses them. It also enables `-parameters` so MapStruct can resolve constructor parameter names.
- `addPoint` carries `@Transactional` because `@ElementCollection` is lazy by default; without an active transaction when reading the list, Hibernate throws `LazyInitializationException`.

### 3. REST API Best Practices

**Versioned base path**

The controller uses `@RequestMapping("/api/v1/blueprints")`. The `/api/v1` prefix is a good versioning practice that allows introducing `/api/v2` in the future without breaking existing clients.

**Correct HTTP codes**

| Situation | Code | Description |
|-----------|------|-------------|
| Successful query | `200 OK` | Successful GET |
| Resource created | `201 Created` | Successful POST |
| Update accepted | `202 Accepted` | Successful PUT |
| Invalid data / duplicate | `400 Bad Request` | Validation failed or blueprint already exists |
| Resource not found | `404 Not Found` | Non-existent author or blueprint |

**Generic `ApiResponse<T>` class**

All responses follow the same JSON contract:

```java
public record ApiResponse<T>(int code, String message, T data) {}
```

Example successful response:
```json
{
  "code": 200,
  "message": "execute ok",
  "data": {
    "author": "john",
    "name": "house",
    "points": [{"x": 0, "y": 0}, {"x": 10, "y": 0}]
  }
}
```

Example 404 error:
```json
{
  "code": 404,
  "message": "Blueprint not found: john/unknown",
  "data": null
}
```

**`GlobalExceptionHandler`**

Error handling is centralized in `GlobalExceptionHandler` with `@RestControllerAdvice`, separating it from the controller:

- `BlueprintNotFoundException` → 404
- `BlueprintPersistenceException` → 400
- `MethodArgumentNotValidException` (`@Valid` failure) → 400 with field detail

Because of this, the controller has no `try-catch` blocks: only business logic.

### 4. OpenAPI / Swagger

The `springdoc-openapi-starter-webmvc-ui` dependency (v2.6.0) is already in `pom.xml`. When the application starts, documentation is automatically available at:

- Swagger UI: [http://localhost:8080/swagger-ui.html](http://localhost:8080/swagger-ui.html)
- OpenAPI JSON: [http://localhost:8080/v3/api-docs](http://localhost:8080/v3/api-docs)

**Configuration (`OpenApiConfig`)**

```java
@Bean
public OpenAPI api() {
    return new OpenAPI().info(new Info()
            .title("ARSW Blueprints API")
            .version("v1")
            .description("Blueprints Laboratory (Java 21 / Spring Boot 3.3.x)"));
}
```

**Controller annotations**

Each endpoint has `@Operation(summary = "...")` and `@ApiResponse(responseCode = "...", description = "...")` to document possible response codes. The name `@ApiResponse` from Swagger conflicts with our `ApiResponse<T>` record, so the full path `@io.swagger.v3.oas.annotations.responses.ApiResponse` is used for the annotation.

```java
@Operation(summary = "Get a specific blueprint by author and name")
@io.swagger.v3.oas.annotations.responses.ApiResponse(responseCode = "200", description = "OK")
@io.swagger.v3.oas.annotations.responses.ApiResponse(responseCode = "404", description = "Blueprint not found")
@GetMapping("/{author}/{bpname}")
public ResponseEntity<ApiResponse<Blueprint>> byAuthorAndName(...) { ... }
```

### 5. Blueprint Filters

**Implementations**

| Filter | Profile | Behavior |
|--------|---------|---------|
| `IdentityFilter` | (none — default) | Returns the blueprint unchanged |
| `RedundancyFilter` | `redundancy` | Removes **consecutive duplicate points** `(x,y)` |
| `UndersamplingFilter` | `undersampling` | Keeps **1 out of every 2 points** (even indices) |

The filter is only applied in `BlueprintsServices.getBlueprint()` (endpoint `GET /{author}/{bpname}`), not in the other endpoints.

**RedundancyFilter example**

Input:  `[(0,0), (0,0), (5,5), (10,10), (10,10)]`  
Output: `[(0,0), (5,5), (10,10)]`

**UndersamplingFilter example**

Input:  `[(0,0), (1,1), (2,2), (3,3), (4,4)]`  
Output: `[(0,0), (2,2), (4,4)]`

**Conflict management between filters**

`IdentityFilter` carries `@Profile("!redundancy & !undersampling")`: it only registers as a bean when no filter profile is active. This avoids `NoUniqueBeanDefinitionException` when combining profiles.

## Available Profile Combinations

**Persistence** profiles (`postgres`, `memory`) and **filter** profiles (`redundancy`, `undersampling`) are independent and can be freely combined.

> In PowerShell, `-D` arguments with dots must be quoted.

```powershell
# PostgreSQL + no filter (default)
mvn spring-boot:run

# PostgreSQL + RedundancyFilter
mvn spring-boot:run "-Dspring-boot.run.profiles=postgres,redundancy"

# PostgreSQL + UndersamplingFilter
mvn spring-boot:run "-Dspring-boot.run.profiles=postgres,undersampling"

# In-memory + no filter
mvn spring-boot:run "-Dspring-boot.run.profiles=memory"

# In-memory + RedundancyFilter
mvn spring-boot:run "-Dspring-boot.run.profiles=memory,redundancy"

# In-memory + UndersamplingFilter
mvn spring-boot:run "-Dspring-boot.run.profiles=memory,undersampling"
```

Start PostgreSQL with Docker before using the `postgres` profile:
```bash
docker run --name blueprints-db \
  -e POSTGRES_PASSWORD=postgres \
  -e POSTGRES_DB=blueprints \
  -p 5432:5432 -d postgres:16
```

## Conclusion

The lab applies three design patterns coherently:

1. **Port/Adapter** — `BlueprintPersistence` is the port; `InMemoryBlueprintPersistence` and `PostgresBlueprintPersistence` are adapters interchangeable by Spring profile. Changing the persistence technology does not modify any other layer.

2. **Strategy** — `BlueprintsFilter` is the strategy; `IdentityFilter`, `RedundancyFilter`, and `UndersamplingFilter` are implementations activatable by profile. Adding a new filter only requires creating a class with `@Profile` and `@Component`.

3. **Uniform response** — `ApiResponse<T>` ensures that all endpoints, both successful and error responses, return the same JSON contract, facilitating consumption by frontend or mobile clients.

The combination of Spring Profiles for persistence and filters allows starting the application in any context (development without a database, production with PostgreSQL, testing with specific filters) without modifying the source code.
